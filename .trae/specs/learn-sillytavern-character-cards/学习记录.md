## 一、什么是角色卡（Character Card）

### 1.1 角色卡的本质

角色卡是一种**特殊格式的 PNG 图片文件**，它在标准 PNG 图片中嵌入了结构化的 JSON 角色数据。简单来说，一张角色卡 = 一张图片 + 一套角色定义数据。当你双击一张角色卡图片时，你看到的是一张普通的角色立绘；但当你将它导入 SillyTavern 时，图片中隐藏的 JSON 数据会被解析出来，构成 AI 角色的完整设定。

### 1.2 技术原理

PNG（Portable Network Graphics）格式支持在文件中存储元数据，具体来说是利用 **PNG tEXt 数据块**。tEXt 块是一种键值对结构的文本元数据，原本用于存储版权信息、作者、软件名称等辅助信息。

角色卡的实现思路如下：

1. 将一个完整的 JSON 对象（包含角色的名称、描述、性格等）序列化为字符串
2. 对该字符串进行 **Base64 编码**（因为 tEXt 块只接受文本内容，Base64 确保二进制安全）
3. 将编码后的字符串作为一个 tEXt 块写入 PNG 文件
4. 读取时反向操作：从 tEXt 块提取 Base64 字符串 → 解码 → 得到 JSON 数据

```
┌─────────────────────────────────┐
│          PNG 文件结构            │
├─────────────────────────────────┤
│  PNG 签名 (8 bytes)             │
│  IHDR 块 (图像头)               │
│  ... 其他图像数据块 ...          │
│  tEXt 块 (关键字: "chara")       │  ← 角色数据存于此
│    └─ Base64 编码的 JSON 字符串  │
│  IEND 块 (图像结束)             │
└─────────────────────────────────┘
```

### 1.3 核心价值

| 价值 | 说明 |
|------|------|
| **一图一角色** | 一张图片即是一个完整的角色，用户直观看到角色外观 |
| **便于分享** | 图片格式通用，可以在任何平台传播、预览，不需要额外文件 |
| **跨平台兼容** | 基于开放标准 PNG + JSON，不依赖特定软件的数据格式 |
| **可扩展性** | JSON 结构允许不断新增字段，版本演进不会破坏兼容性 |

### 1.4 SillyTavern 与角色卡的关系

**SillyTavern** 是一个开源的 AI 角色扮演前端（前端界面），它将用户连接到各种 LLM 后端（如 OpenAI API、KoboldAI、TextGen WebUI、NovelAI 等），提供丰富的角色扮演功能。

角色卡是 SillyTavern 的**核心数据载体**：

- SillyTavern 通过解析角色卡的 JSON 数据来构建发送给 LLM 的提示词（Prompt）
- 角色卡中的 `description`、`personality`、`scenario` 等字段会被组装成"角色定义"，嵌入到每一次对话请求中
- 角色卡支持嵌入 **Lorebook（角色知识库）**，在对话中通过关键词触发注入额外的世界设定信息
- SillyTavern 支持导入/导出多种格式的角色卡，并提供内置编辑器来创建和修改角色

## 二、角色卡规范的版本演进

角色卡规范经历了三个主要版本的迭代，每个版本都在前一个版本的基础上增加了新的能力。

### 2.1 V1：基础版本

V1 是最早的角色卡格式，定义了 **6 个核心字段**：

| 字段 | 说明 |
|------|------|
| `name` | 角色名称 |
| `description` | 角色描述（外貌、背景等） |
| `personality` | 角色性格特质 |
| `scenario` | 互动场景设定 |
| `first_mes` | 角色的首条消息 |
| `mes_example` | 对话示例 |

V1 没有 `spec` 和 `spec_version` 字段，结构简单直观，为后续发展奠定了基础。早期角色卡领域的大量社区创作都基于 V1 格式。

### 2.2 V2：当前主流标准

**V2 是目前最广泛使用的版本**，它引入了明确的规范标识和大量新字段。

标识信息：
- `spec`: `"chara_card_v2"`
- `spec_version`: `"2.0"`
- PNG 中使用 tEXt 关键字 `"chara"`

**V2 新增字段**：

| 字段 | 说明 |
|------|------|
| `creator_notes` | 创作者备注，不参与提示词构建 |
| `system_prompt` | 角色专属系统提示词，支持 `{{original}}` 占位符 |
| `post_history_instructions` | 历史后指令（替代 UJB/Jailbreak） |
| `alternate_greetings` | 多个可选开场白（字符串数组） |
| `character_book` | 嵌入式 Lorebook，角色专属知识库 |
| `tags` | 分类标签数组 |
| `creator` | 作者名 |
| `character_version` | 角色版本号 |
| `extensions` | 扩展数据（默认 `{}`） |

### 2.3 V3：扩展格式

V3 在 V2 的基础上进一步扩展，支持更丰富的媒体资源。

标识信息：
- `spec`: `"chara_card_v3"`
- `spec_version`: `"3.0"`
- PNG 中使用 tEXt 关键字 `"ccv3"`

**V3 新增字段**：

| 字段 | 说明 |
|------|------|
| `group_only_greetings` | 群聊专用开场白 |
| `regex_scripts` | 正则脚本（用于自动化文本处理） |
| `assets` | 资产引用，支持多文件嵌入 |

**CharX 压缩包格式**：V3 引入了 `.charx` 格式，本质上是一个 **ZIP 压缩包**，内部可以包含：
- 一个或多个角色立绘图片
- 视频文件
- 其他媒体资产
- 主 JSON 数据文件

这使得角色卡不再局限于"一张图的 PNG"，而是可以成为一个完整的多媒体角色包。

### 2.4 版本对比总览

| 特性 | V1 | V2 | V3 |
|------|:--:|:--:|:--:|
| `spec` 标识 | ❌ | `chara_card_v2` | `chara_card_v3` |
| `spec_version` | ❌ | `2.0` | `3.0` |
| PNG tEXt 关键字 | 无规范 | `chara` | `ccv3` |
| 核心 6 字段 | ✅ | ✅ | ✅ |
| `system_prompt` | ❌ | ✅ | ✅ |
| `post_history_instructions` | ❌ | ✅ | ✅ |
| `character_book` | ❌ | ✅ | ✅ |
| `alternate_greetings` | ❌ | ✅ | ✅ |
| `tags` / `creator` / `character_version` | ❌ | ✅ | ✅ |
| `.charx` 压缩包 | ❌ | ❌ | ✅ |
| `group_only_greetings` | ❌ | ❌ | ✅ |
| 多图片/视频/资产 | ❌ | ❌ | ✅ |

## 三、V2 核心字段详解

> 以 V2 规范为主线，这是目前最主流且功能完备的版本。所有字段说明适用于实际创作。

### 3.1 name

- **类型**：`string`
- **Token 类别**：永久 Token
- **说明**：角色的名称。保持简洁，通常是一个词或短名。

AI 在生成消息时会使用这个名字来标注角色发言。过长的名称会浪费 Token 预算。

```json
"name": "Elara"
```

### 3.2 description

- **类型**：`string`
- **Token 类别**：永久 Token（始终在提示词中）
- **说明**：描述角色的**外貌特征、背景故事、世界观设定**。这是 AI 理解"这个角色是谁、长什么样、来自哪里"的核心依据。

description 在每次生成请求时都会被发送给 LLM，因此其 Token 消耗影响最大。应**只写外貌和背景设定**，不写内心性格（放在 personality 中）。

```json
"description": "Elara is a 24-year-old elven archer from the Silverwood Forest. She has long silver hair, emerald green eyes, and pointed ears. She wears a dark green cloak and carries an ornate longbow passed down through her family for generations. The Silverwood is a mystical forest on the edge of the human kingdom, home to the last remaining elf clans."
```

### 3.3 personality

- **类型**：`string`
- **Token 类别**：永久 Token（始终在提示词中）
- **说明**：描述角色的**内在性格、动机、价值观、言行风格**。告诉 AI "这个角色如何思考和行动"。

personality 应专注于内在特质，不要写外貌信息（那是 description 的职责）。可以包含说话方式、态度、情感倾向、行为模式等。

```json
"personality": "Elara is calm and observant, often speaking in a measured, thoughtful tone. She is fiercely protective of nature and her people, but carries a deep sadness from the gradual decline of the elf clans. She is slow to trust humans but will show warmth once trust is earned. Her humor is dry and subtle, and she prefers action over words."
```

### 3.4 scenario

- **类型**：`string`
- **Token 类别**：永久 Token（始终在提示词中）
- **说明**：设定**互动的场景上下文**，包括时间、地点、初始情境、相遇原因等。

scenario 帮助 AI 理解"当前发生了什么、角色和用户为什么在一起"。

```json
"scenario": "{{char}} and {{user}} meet at the edge of the Silverwood Forest at dusk. {{user}} is a wandering adventurer seeking the help of the elf clans. A recent attack from shadow creatures has left {{char}} suspicious of outsiders."
```

### 3.5 first_mes

- **类型**：`string`
- **Token 类别**：临时 Token（仅在聊天开始时发送一次）
- **说明**：**角色说的第一句话**。这是决定角色交流风格和语气的关键元素——用户对角色形成第一印象的核心时刻。

first_mes 不占用后续对话的 Token 预算，因此可以写得较为详细。一个好的 first_mes 应该：
- 展现角色的说话风格和语气
- 设定最初的互动情境
- 给用户提供明确的回应方向

```json
"first_mes": "*The soft rustle of leaves announces her presence before her voice does. Elara steps out from behind an ancient oak, her bow lowered but her eyes watchful.*\n\n\"You're a long way from the trade roads, stranger. Few wander this deep into the Silverwood without purpose. State yours, and choose your words carefully — the forest listens.\""
```

### 3.6 mes_example

- **类型**：`string`
- **Token 类别**：临时 Token（随上下文填满可能被推出）
- **说明**：**对话示例**，用于向 AI 展示角色的对话风格、节奏和行为模式。使用 `<START>` 作为每个示例块的分隔符。

mes_example 不占用永久 Token，但随着对话历史增长，它可能被从上下文中推出。因此应保持**精炼、有代表性**。

```json
"mes_example": "<START>\n{{user}}: What brings you to this part of the woods?\n{{char}}: *Elara nocks an arrow with practiced ease but doesn't draw.* \"Protection. The Silverwood has stood for millennia. I intend to see it stand for millennia more. The question is — what brings *you* here?\"\n<START>\n{{user}}: Can you teach me to shoot like that?\n{{char}}: *She lets out a soft, almost imperceptible laugh.* \"No one learns to shoot in a day, outsider. But if you're willing to bleed for it...\" *She gestures toward a nearby clearing.* \"We start at dawn.\""
```

### 3.7 creator_notes

- **类型**：`string`
- **Token 类别**：**不参与提示词构建**
- **说明**：创作者为自己或其他创作者写的备注。可以包含创作意图、灵感来源、使用建议等。LLM 永远不会看到此字段的内容。

```json
"creator_notes": "Elara is designed for a slow-burn fantasy adventure. She works best with GPT-4 class models. Lower temperature (0.7-0.9) recommended to maintain her measured tone."
```

### 3.8 system_prompt

- **类型**：`string`
- **Token 类别**：永久 Token（如果设置）
- **说明**：**角色专属的系统提示词**，用于覆盖或扩展用户的默认系统提示词。支持 `{{original}}` 占位符，该占位符会被替换为用户在 SillyTavern 设置中配置的默认系统提示词。

这使得角色创作者可以为角色定制行为约束，而不必依赖于用户的全局设置。

```json
"system_prompt": "{{original}}\n\n[IMPORTANT: You are roleplaying as Elara, an elven archer. You should never break character. Your responses should always include subtle physical descriptions of your movements and the forest environment around you.]"
```

### 3.9 post_history_instructions

- **类型**：`string`
- **Token 类别**：永久 Token（如果设置）
- **说明**：**历史后指令**（Post-History Instructions），在对话历史之后、AI 生成回复之前插入。这替代了早期的 UJB（Universal Jailbreak）概念。

同样支持 `{{original}}` 占位符。用于在 AI 生成回复的最后一刻施加行为指导。

```json
"post_history_instructions": "{{original}}\n\n[Remember: Elara does not use modern slang. Her speech is formal but not archaic. She often pauses to observe her surroundings mid-conversation. Do not speak for {{user}}.]"
```

### 3.10 alternate_greetings

- **类型**：`string[]`（字符串数组）
- **Token 类别**：临时 Token（仅选中的那条发送）
- **说明**：提供**多个可选的开场白**。用户在开始新对话时可以从这些候选中选择一个。

对于希望在不同场景中使用同一角色的情况非常有用（如：初次见面、熟悉后重逢、敌对状态等不同起始情境）。

```json
"alternate_greetings": [
  "*Elara stands at the edge of the Silverwood, bow in hand, watching you approach.* \"State your business, traveler.\"",
  "*In the warmth of a campfire, Elara looks across the flames at you.* \"You know, you're the first human I've trusted with my camp in fifty years.\"",
  "*Elara emerges from the shadows, arrow aimed at your heart.* \"I warned you never to return.\""
]
```

### 3.11 character_book

- **类型**：`object`（CharacterBook 结构）
- **Token 类别**：条件 Token（通过关键词触发注入）
- **说明**：**嵌入式 Lorebook**，角色专属的世界知识库。详细结构见[第七章](#七lorebook角色知识库)。

```json
"character_book": {
  "name": "Silverwood Lore",
  "description": "Lore entries for Elara's world",
  "scan_depth": 50,
  "token_budget": 500,
  "recursive_scanning": false,
  "entries": [
    {
      "keys": ["Silverwood", "forest"],
      "content": "The Silverwood is an ancient mystical forest...",
      "enabled": true,
      "insertion_order": 100
    }
  ]
}
```

### 3.12 tags

- **类型**：`string[]`（字符串数组）
- **Token 类别**：**不参与提示词构建**
- **说明**：用于分类和搜索的标签。用于在角色库中筛选角色，不会发送给 LLM。

```json
"tags": ["fantasy", "elf", "archer", "adventure", "slow-burn", "female"]
```

### 3.13 creator

- **类型**：`string`
- **Token 类别**：**不参与提示词构建**
- **说明**：角色卡作者的名字或化名。

```json
"creator": "AliceTheWriter"
```

### 3.14 character_version

- **类型**：`string`
- **Token 类别**：**不参与提示词构建**
- **说明**：角色的版本号，便于跟踪迭代更新。

```json
"character_version": "1.2.0"
```

### 3.15 extensions

- **类型**：`object`
- **Token 类别**：取决于扩展类型
- **说明**：扩展数据容器，默认值为 `{}`。用于存储第三方插件或未来扩展定义的额外数据。

```json
"extensions": {}
```

## 四、Token 预算管理

Token 预算是角色卡制作中**最容易被忽视但最重要的技术考量**。合理管理 Token 直接决定了 AI 角色扮演的质量和对话长度。

### 4.1 Token 分类

#### 永久 Token（Permanent Tokens）

这些字段的内容在**每次生成请求中都会被发送给 LLM**：

| 字段 | 说明 |
|------|------|
| `name` | 角色名称（通常 < 5 tokens） |
| `description` | 外貌、背景、世界设定 |
| `personality` | 内在性格、动机、言行风格 |
| `scenario` | 互动场景上下文 |
| `system_prompt`（如果设置） | 角色专属系统提示词 |
| `post_history_instructions`（如果设置） | 历史后指令 |

**关键影响**：永久 Token 越多，可用于保存聊天历史的 Token 越少，AI 的"记忆力"越差。

#### 临时 Token（Temporary Tokens）

这些字段仅在特定条件下发送，不持续占用预算：

| 字段 | 发送时机 |
|------|----------|
| `first_mes` | 仅在聊天开始时发送一次 |
| `mes_example` | 随上下文填充，对话增长后可能被逐出 |

### 4.2 推荐 Token 预算

| 区间 | Token 数 | 效果 |
|------|----------|------|
| **理想范围** | 500 ~ 2200 | 角色信息充足，AI 记忆良好 |
| **信息不足** | < 250 | 角色定义过于简单，AI 表现不稳定 |
| **信息过多** | > 2700 | AI "记忆力"显著下降，聊天历史保存更少 |
| **红色警报** | > 模型上下文一半 | 严重挤占对话历史空间 |

### 4.3 各流行模型上下文限制

| 模型 | 最大上下文 | 推荐永久 Token 上限 |
|------|-----------|---------------------|
| LLaMA 3 (8B) | 8,192 | ~2,000 |
| LLaMA 3 (70B) | 8,192 | ~2,000 |
| GPT-3.5 Turbo | 16,385 | ~4,000 |
| GPT-4 / GPT-4 Turbo | 128,000 | ~5,000 |
| Claude 3 (Haiku/Sonnet/Opus) | 200,000 | ~5,000 |
| Command R+ | 128,000 | ~5,000 |

> 永久 Token 上限为推荐值，实际可根据需求调整。但永远不要让角色定义 Token 超过模型总上下文的 **50%**。

### 4.4 Token 计算工具

在实际创作中，可以使用以下工具估算 Token：
- **OpenAI Tokenizer**（platform.openai.com/tokenizer）：精确计算 GPT 系列模型的 Token 数
- **SillyTavern 内置显示**：角色编辑界面通常会显示 Token 统计
- **tiktoken 库**：编程方式计算（Python）

## 五、角色描述写作格式

角色卡的 `description` 和 `personality` 字段可以使用多种写作格式。不同格式在 Token 效率、可读性和 AI 理解程度上有各自的特点。

### 5.1 自由文本（Free Text）

**最自然、最直观的格式**，使用自然语言段落描述角色。

```text
Elara is a 24-year-old elven archer from the Silverwood Forest. She has long silver hair, emerald green eyes, and wears a dark green cloak. She is calm and observant, fiercely protective of nature and her people. She speaks in a measured tone and is slow to trust humans.
```

**优点**：
- 自然流畅，易于人类阅读和编辑
- AI 理解自然语言非常出色
- 无学习成本

**缺点**：
- Token 效率较低（描述性词语占用较多 Token）
- 结构化程度低

### 5.2 W++ 格式

**结构化属性标签格式**，使用方括号包裹属性类别和值。格式为 `[Category(attribute1, attribute2, ...)]`。

```text
[Appearance(long silver hair, emerald green eyes, pointed ears, dark green cloak, ornate longbow)]
[Clothing(dark green cloak, leather bracers, soft boots)]
[Personality(calm, observant, protective, slow to trust, dry humor)]
[Motivation(protect the Silverwood, preserve elf heritage, find lost clan members)]
[Speech(measured, thoughtful, formal but not archaic, pauses often)]
[Background(24 years old, Silverwood Forest native, last surviving member of her scout group)]
```

**优点**：
- Token 效率高（紧凑的键值结构）
- 高度结构化，便于 AI 解析

**缺点**：
- 格式较机械，缺乏温度
- 需要学习 W++ 规范

### 5.3 Boostyle 格式

**关键词/属性列表格式**，极度紧凑，使用少量词汇密集描述。通常用加号连接或用逗号分隔。

```text
Elara[elf + female + 24yo + archer + Silverwood + silver hair + emerald eyes + calm + observant + protective + dry humor + measured speech + formal + slow to trust + forest guardian + longbow + dark green cloak]
```

或者更简洁的纯关键词列表：

```text
elf, female, 24yo, archer, Silverwood, silver hair, emerald eyes, calm, observant, protective, dry humor, measured speech, formal, slow to trust
```

**优点**：
- Token 效率极高
- 适用于上下文窗口较小的模型

**缺点**：
- 可读性差
- AI 可能无法从离散关键词中构建连贯的角色形象

### 5.4 Ali:Chat 格式

Ali:Chat 是一种哲学层面的写作方法论，其**核心思想是通过对话示例来表达角色特质，而非在 description 中平铺直述**。

#### 核心理念

传统方法是在 description 中写"Iron Bull 很大声且粗鲁"，Ali:Chat 则认为应该**通过对话展示这一点**。

#### 四个关键原则

**1. description 中包含对话**

在 description 中嵌入角色的对话片段（非正式对话，而是角色心理/态度的展现）：

```text
Elara is an elf archer. {{char}} sometimes thinks, "These humans trample through the forest like deaf bears — how did their species survive this long?"
```

这比写 `Elara dislikes humans who disrespect nature` 更生动。

**2. first_mes 详细设定场景**

first_mes 不仅是问候，而是**建立整个场景的情绪基调、空间感和互动理由**：

```text
*The Silverwood at dusk is a cathedral of silence, broken only by the distant hoot of an owl. Elara has been tracking {{user}} for the past hour, her footsteps silent on the moss-covered earth. Finally, she steps into view, positioning herself between {{user}} and the deeper woods.*

"You've been wandering in circles for an hour, outsider. The forest doesn't want you here yet. Why should I feel differently?"
```

**3. mes_example 推动对话方向**

mes_example 不只是展示说话风格，而是**示范角色如何推进互动**：

```text
<START>
{{user}}: I'm lost. Can you help me find the road?
{{char}}: *Elara studies {{user}} for a long moment, then gestures north.* "The road is that way." *She doesn't move.* "But you didn't come to the Silverwood just to find a road, did you? No one does."
```

**4. 不重复信息**

如果 personality 已经写了"calm"，description 不需要再写一遍。每个字段做自己该做的事，避免冗余浪费 Token。

## 六、角色卡质量评分标准与最佳实践

### 6.1 五大评分维度（各 0–20 分）

| 维度 | 满分 | 评估内容 |
|------|:----:|----------|
| **1. 字段适配性与结构** | 20 | 是否正确使用各字段，不混用职责 |
| **2. 角色连贯性与一致性** | 20 | description、personality、scenario、first_mes 是否描述同一角色 |
| **3. 角色扮演实用性与沉浸感** | 20 | first_mes 是否设定语调、场景和互动理由 |
| **4. 创意与原创性** | 20 | 构思是否有新意，避免陈词滥调 |
| **5. 打磨与技术要求** | 20 | 语法清晰，格式可读，正确使用 `{{char}}`/`{{user}}` 占位符 |
| **总分** | **100** | *(注：原始评分表有 5 分的额外加分空间，满分实为 105)* |

### 6.2 等级划分

| 等级 | 分数区间 | 含义 |
|:----:|----------|------|
| **S** | 90–105 | 卓越：角色在所有方面均表现出色 |
| **A** | 80–89 | 优秀：高质量角色，体验很好 |
| **B** | 60–79 | 良好：可用，但有几个明显改进空间 |
| **C** | 40–59 | 及格：有基本功能但体验平平 |
| **D** | 30–39 | 差：存在重大问题 |
| **F** | 0–29 | 极差：基本不可用 |

### 6.3 常见错误

| 错误类型 | 表现 | 正确做法 |
|----------|------|----------|
| **字段空置或误用** | personality 为空，或把性格写进了 description | 每个字段各司其职（见下方"字段内容分配建议"） |
| **缺少 first_mes** | 没有 first_mes，或仅写 `"Hi"`、`"Hello"` | 撰写有场景感、展现性格的开场白 |
| **无 mes_example** | mes_example 字段为空 | 至少提供 2–3 个交错的对话示例 |
| **单个字段过度填充** | description 写了 5000 字 | 控制 Token，把详细信息放入 Lorebook |
| **性格矛盾** | personality 写"温柔"，first_mes 写攻击性内容 | 所有字段保持角色一致 |
| **Token 总量失衡** | 永久 Token 超过 3000 | 精简到 500–2200 的理想范围 |

### 6.4 字段内容分配建议

明确每个字段应该写什么、**不应该写什么**，是避免常见错误的关键：

| 字段 | 应该写 | 不应该写 |
|------|--------|----------|
| **description** | 外貌特征、背景故事、世界设定 | 内心情感、性格特质（应放在 personality） |
| **personality** | 内在特质、社交风格、动机、价值观 | 外貌描述（应放在 description） |
| **scenario** | 时间、地点、相遇原因、初始情势 | 详细的角色信息（应在其他字段） |
| **first_mes** | 强有力的开场白，展现角色风格和场景 | 平淡的问候语如 "Hello" |
| **mes_example** | 短小精悍的对话片段，用 `<START>` 分隔 | 长篇叙事（应保持示例精简） |

### 6.5 最佳实践清单

- ✅ 使用 `{{char}}` 和 `{{user}}` 占位符代替硬编码的角色名和用户名
- ✅ 每个字段只做自己的事，不混用职责
- ✅ first_mes 中融入动作和场景描述（用星号包裹 `*like this*`）
- ✅ mes_example 提供 2–5 个简短但有代表性的对话片段
- ✅ 将大量世界设定细节放入 Lorebook 而非 description
- ✅ 定期检查 Token 总数，保持在 500–2200 的理想范围
- ✅ 添加合理数量的 tags，帮助用户搜索
- ✅ 为不同起始场景提供 alternate_greetings

## 七、Lorebook（角色知识库）

### 7.1 概念

**Lorebook**（也称为 World Book 或 Character Book）是一个嵌入在角色卡中的**世界设定知识库**。它由多个条目（Entry）组成，每个条目定义了一组触发关键词和相关内容。当对话中出现关键词时，对应的内容会被动态注入到 LLM 的上下文中。

与永久 Token 不同，Lorebook 的内容**仅在触发时才注入**，因此它是管理大量世界设定细节的高效方式。

### 7.2 条目（Entry）结构

每个 Lorebook 条目的基本结构：

```json
{
  "keys": ["Silverwood", "forest", "ancient woods"],
  "content": "The Silverwood is an ancient mystical forest on the edge of the human kingdom...",
  "enabled": true,
  "insertion_order": 100
}
```

| 属性 | 类型 | 说明 |
|------|------|------|
| `keys` | `string[]` | 触发关键词数组。当对话中出现任一关键词时，条目被注入 |
| `content` | `string` | 注入到上下文中的具体内容 |
| `enabled` | `boolean` | 条目是否启用 |
| `insertion_order` | `number` | 插入顺序编号（决定多个条目在上下文中的排列顺序） |

### 7.3 高级属性

| 属性 | 类型 | 说明 |
|------|------|------|
| `selective` | `boolean` | 双重关键词触发 —— 条目的 keys 需要对话中**同时出现至少两个**关键词才触发 |
| `constant` | `boolean` | 始终注入 —— 无论对话内容如何，该条目总是被包含在上下文中 |
| `recursive_scanning` | `boolean` | 递归触发 —— 条目注入的内容本身也可以触发其他条目 |
| `position` | `string` | 注入位置 —— `"before_char"`（角色定义之前）或 `"after_char"`（角色定义之后） |
| `priority` | `number` | 优先级 —— 当 Token 预算不足时，**低优先级的条目先被丢弃** |

### 7.4 CharacterBook 整体结构

```json
"character_book": {
  "name": "Silverwood Encyclopedia",
  "description": "Comprehensive lore for Elara's world",
  "scan_depth": 50,
  "token_budget": 500,
  "recursive_scanning": false,
  "entries": [
    {
      "keys": ["Silverwood", "forest"],
      "content": "The Silverwood is an ancient mystical forest covering the northeastern border of the human kingdom. The trees here are thousands of years old, their bark shimmering with a faint silver glow under moonlight. It is home to the last remaining elf clans.",
      "enabled": true,
      "insertion_order": 100,
      "priority": 10
    },
    {
      "keys": ["elf clans", "elves", "clans"],
      "content": "The elf clans of the Silverwood are divided into three major houses: House Aldari (archers and scouts), House Mithrendil (scholars and mages), and House Thornwood (warriors and guardians). Elara belongs to House Aldari.",
      "enabled": true,
      "insertion_order": 200,
      "priority": 8
    },
    {
      "keys": ["shadow creatures", "darkness", "attack"],
      "content": "Shadow creatures are malevolent beings that emerged from the deep caverns beneath the Silverwood fifty years ago. They are vulnerable to light and enchanted silver. The elf clans have been at war with them ever since, slowly losing ground.",
      "enabled": true,
      "insertion_order": 300,
      "priority": 7
    }
  ]
}
```

| CharacterBook 属性 | 说明 |
|-------------------|------|
| `name` | 知识库名称 |
| `description` | 知识库描述（不发送给 LLM） |
| `scan_depth` | 扫描对话历史的深度（最近 N 条消息） |
| `token_budget` | Lorebook 注入内容的最大 Token 预算 |
| `recursive_scanning` | 是否启用递归触发 |
| `entries` | Lorebook 条目数组 |

### 7.5 使用策略

- **将细节从 description 移入 Lorebook**：如果某个世界设定细节不会在每一轮对话中都用到，它就应该在 Lorebook 中
- **关键词设计要自然**：触发关键词应该是在对话中自然出现的词，而非生僻的标记
- **控制条目大小**：每个条目应简短精炼（50–150 tokens），不要写成长篇大论
- **合理设置优先级**：核心设定给高优先级，边缘细节给低优先级
- **token_budget 要留有空间**：不要设得过高，确保有足够的 Token 留给对话

## 十、宏变量与占位符

宏变量（Macros）是角色卡中使用的特殊占位符，它们在发送给 LLM 之前会被替换为实际值。掌握宏变量是写出通用、可复用角色卡的关键。

### 10.1 名称类宏

| 宏 | 替代写法 | 替换为 |
|-----|----------|--------|
| `{{char}}` | `<BOT>` | 角色的 `name` 字段值 |
| `{{user}}` | `<USER>` | 用户在 SillyTavern 中设置的用户名 |

```text
// 角色卡中写：
{{char}} looks at {{user}} with curiosity.

// 实际发送给 LLM：
Elara looks at 小明 with curiosity.
```

> **最佳实践**：始终使用 `{{char}}` 和 `{{user}}` 代替硬编码名称。这样角色卡可以在不同用户间复用，而不需要每个用户手动修改名称。

### 10.2 占位替换类宏

| 宏 | 替换为 |
|-----|--------|
| `{{original}}` | 用户在 SillyTavern 设置中配置的原始系统提示词 / Jailbreak 文本 |
| `{{charPrompt}}` | 角色的主要提示覆盖（Character Prompt Override） |
| `{{charJailbreak}}` | 角色的 Jailbreak 覆盖 |

`{{original}}` 是最重要的占位符宏，通常用于 `system_prompt` 和 `post_history_instructions` 字段：

```json
"system_prompt": "{{original}}\n\n[Always stay in character as Elara.]"
```

这样设置后，角色的专属指令会**追加**在用户的默认系统提示词之后，而不是完全覆盖它。

### 10.3 内容类宏

| 宏 | 替换为 |
|-----|--------|
| `{{description}}` | 角色的 `description` 字段内容 |
| `{{personality}}` | 角色的 `personality` 字段内容 |
| `{{scenario}}` | 角色的 `scenario` 字段内容 |

这些宏允许在提示词模板中灵活引用角色字段的内容。它们通常在提示词模板中使用，而不是在角色卡字段本身中使用。

### 10.4 格式化宏

| 宏 | 效果 |
|-----|------|
| `{{newline}}` | 插入一个换行符（`\n`） |
| `{{trim}}` | 修剪（删除）包围此宏的换行符，防止产生多余空行 |
| `{{noop}}` | 无操作，回退为空字符串。用于占位或注释目的 |

```text
// 使用示例：
第一段文字 {{trim}}
{{newline}}
第二段文字

// 效果：{{trim}} 消除了前后的多余换行，得到一个整洁的换行连接。
```

### 10.5 宏变量使用注意事项

- ✅ 宏变量**区分大小写**，`{{char}}` 和 `{{CHAR}}` 是不同的
- ✅ 在 `first_mes` 和 `mes_example` 中必须使用宏变量而非硬编码名称
- ✅ `{{original}}` 只能用于 `system_prompt` 和 `post_history_instructions`
- ✅ `{{trim}}` 对于控制提示词格式和节省 Token（减少不必要的空行）非常有用
- ⚠️ 宏变量**不会递归展开**——一个宏变量的值中如果包含另一个宏变量的语法，不会被再次替换
